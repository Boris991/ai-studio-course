## Теоретический материал

### От инструментов к агентскому циклу

В предыдущих разделах мы строили агентов, которые вызывали инструменты в ответ на один конкретный запрос. Однако для решения сложных исследовательских задач одного шага недостаточно: агент должен **планировать**, **выполнять несколько действий**, **анализировать промежуточные результаты** и **принимать решения** о дальнейших шагах.

Такой процесс называется **агентский цикл** (agentic loop), и агент, его реализующий, часто именуется **ReAct-агентом** (от Reasoning and Acting):

1. Агент получает задачу и **планирует**, на какие подзадачи её разбить.
2. Выбирает **инструмент** для текущей подзадачи.
3. **Выполняет действие** (вызывает инструмент).
4. **Анализирует результат** и решает, достаточно ли информации.
5. **Повторяет цикл**, пока задача не выполнена.

Количество итераций не фиксировано — модель самостоятельно решает, когда остановиться. Это принципиально отличает agнтский цикл от простого однократного вызова инструмента.

### OpenAI Agents SDK

Библиотека [OpenAI Agents SDK](https://github.com/openai/openai-agents-python) предоставляет высокоуровневый фреймворк для создания как одиночных, так и многоагентных систем на основе моделей, поддерживающих Responses API. Для работы с Yandex Cloud используются:

- **`AsyncOpenAI`** — асинхронный клиент для работы с Responses API. Асинхронность позволяет не блокировать выполнение локального цикла во время ожидания ответа модели.
- **`OpenAIResponsesModel`** — обёртка, инкапсулирующая клиент и название модели для использования в Agents SDK.
- **`Agent`** — класс агента, определяющий системный промпт, набор инструментов и связи с другими агентами.
- **`Runner.run()`** — запускает агентский цикл, в котором агент автоматически вызывает инструменты и обрабатывает результаты до получения финального ответа.
- **`@function_tool`** — декоратор для создания инструментов из обычных Python-функций. Документация функции (docstring) и аннотации типов автоматически преобразуются в описание инструмента для LLM.
- **`WebSearchTool()`** — встроенный инструмент для поиска в интернете.

> **Важно:** для работы с Yandex Cloud необходимо отключить трейсинг OpenAI (`set_tracing_disabled(True)`), так как он требует ключа OpenAI API.

### Мониторинг: RunHooks

При выполнении длительных исследований полезно отслеживать действия агента в реальном времени. Для этого можно использовать механизм **RunHooks** — класс с коллбэками, которые вызываются при наступлении различных событий:

| Событие | Метод | Описание |
|---------|-------|----------|
| Запуск агента | `on_agent_start` | Агент начал работу |
| Завершение агента | `on_agent_end` | Агент завершил работу, доступен результат |
| Вызов инструмента | `on_tool_start` | Агент решил вызвать инструмент |
| Результат инструмента | `on_tool_end` | Инструмент вернул результат |
| Передача управления | `on_handoff` | Управление передано другому агенту |
| Завершение LLM-вызова | `on_llm_end` | LLM вернула ответ (может содержать вызовы hosted tools) |

Через `on_llm_end` можно отслеживать вызовы **hosted tools** (например, `web_search_call`), которые выполняются на стороне API и не проходят через обычные `on_tool_start`/`on_tool_end`.

### Мониторинг: Streaming Events

Альтернативный подход к мониторингу — **стриминг событий** через `Runner.run_streamed()`. В отличие от Hooks, стриминг позволяет обрабатывать события в едином асинхронном цикле `async for` и, что критически важно, **видеть генерацию текста в реальном времени**, токен за токеном.

Три типа событий, которые возвращаются в цикле стриминга:

| Тип | Описание | Применение |
|-----|----------|------------|
| `raw_response_event` | Низкоуровневые токены от LLM | Отображение текста в реальном времени |
| `run_item_stream_event` | Вызовы и результаты инструментов | Отслеживание прогресса |
| `agent_updated_stream_event` | Смена агента (handoff) | Отображение передачи управления |

Стриминг особенно полезен для пользовательских интерфейсов, где нужно показывать ответ по мере генерации — аналогично тому, как это работает в ChatGPT или Deep Research от OpenAI.

### Многоагентные системы: зачем нужно несколько агентов

Использование нескольких взаимодействующих агентов вместо одного обосновано несколькими причинами:

1. **Ограничение по инструментам.** При наличии большого количества инструментов (10+) точность их вызова падает. Специализированные агенты с 2-3 инструментами работают значительно лучше.

2. **Разные точки зрения.** Благодаря различным системным промптам разные агенты «смотрят» на задачу по-разному. Например, при медицинской диагностике координирующий агент-терапевт может передавать случай агентам-специалистам.

3. **Декомпозиция задачи.** Дополнительный уровень разбиения сложной задачи на подзадачи упрощает работу для каждого отдельного агента, повышая общее качество решения.

### Паттерны координации многоагентных систем

OpenAI Agents SDK поддерживает различные механизмы координации агентов в многоагентной системе: передача управления (handoff), агенты-как-инструменты. Это позволяет строить различные паттерны координации агентов, каждый из которых подходит для разных сценариев.

Мы рассмотрим паттерны на примере многоагентной системы Deep Research, в котором будут разделены следующие роли:

* **Planner** - планировщик исследования, разбивающий сложный запрос на фрагменты
* **Searcher** - поисковик, который ищет статьи в интернет и сохраняет их краткое содержание в заметках
* **Writer** - агент, который пишет финальный ответ

#### Sequential Chain (Последовательная цепочка)

```
User → Planner → Searcher → Writer → User
```

Агенты образуют **последовательную цепочку**: каждый передаёт управление следующему через механизм **handoff**. Handoff — это операция, похожая на вызов инструмента: LLM решает, что текущий этап завершён, и передаёт управление указанному агенту. При этом **вся история диалога** передаётся между агентами — каждый видит всю предыдущую переписку.

**Особенность:** агент, начавший цепочку, не получает контроль обратно — финальный агент отвечает пользователю. Агенты определяются в **обратном порядке** (от конца к началу), так как каждый должен знать, кому передать управление.

**Когда использовать:** конвейерные задачи с фиксированной последовательностью этапов.

#### Hub-and-Spoke (Координатор)

```
        ┌──→ Planner ──┐
User → Coordinator ←────┤
        ├──→ Searcher ─┤
        └──→ Writer ───┘
```

Центральный **координатор** управляет процессом: получает задачу, решает, какому агенту передать управление, получает результат, и снова принимает решение о следующем шаге. Все sub-агенты возвращают управление координатору через обратные handoff-ы.

Для реализации циклических зависимостей (координатор → агент → координатор) агент-координатор сначала создаётся как заглушка, а затем наполняется инструкциями и связями.

**Особенность:** координатор сам выстраивает цепочку вызовов в зависимости от задачи. Вся история видна всем агентам. Достаточно интеллектуальные модели способны строить разумные цепочки без детальных инструкций.

**Когда использовать:** динамические workflow, когда порядок действий зависит от промежуточных результатов.

#### Agents-as-Tools (Агенты как инструменты)

```
User → Coordinator ─┬─ call planner_tool() → result
                    ├─ call searcher_tool() → result
                    └─ call writer_tool() → result
                    → Final response
```

Sub-агенты оборачиваются в инструменты через `agent.as_tool()` и вызываются координатором как обычные функции. Координатор получает результат каждого агента как строку и полностью контролирует поток выполнения.

**Особенность:** sub-агенты **не видят общую историю** — каждый получает новый контекст только из переданного координатором промпта. Это обеспечивает изоляцию, но ограничивает стриминг событий от sub-агентов.

**Когда использовать:** сложная оркестрация, когда нужна изоляция контекстов и полный контроль координатора.

### Сравнение паттернов

| Паттерн | Контроль | Контекст | Стриминг | Когда использовать |
|---------|----------|----------|----------|-------------------|
| **Sequential Chain** | Последний агент | Полная передача | ✅ Полный | Простые конвейеры |
| **Hub-and-Spoke** | LLM-координатор | Полная передача | ✅ Полный | Динамические workflow |
| **Agents-as-Tools** | LLM-координатор | Новый контекст | ⚠️ Только верхний уровень | Сложная оркестрация |
